## テストの種類

本プロジェクトでは **ユニットテスト** と **E2E テスト** の 2 種類のテストを運用します。

| 種類           | 概要                                                                    |
| -------------- | ----------------------------------------------------------------------- |
| ユニットテスト | コンポーネント・Context・認証ロジックを Vitest + Testing Library で検証 |
| E2E テスト     | Playwright でブラウザ操作とネットワークを含めて結合検証する             |

---

## 使用フレームワーク

| ツール                                                                            | 用途                               |
| --------------------------------------------------------------------------------- | ---------------------------------- |
| [Vitest](https://vitest.dev/)                                                     | テストランナー・アサーション       |
| [React Testing Library](https://testing-library.com/)                             | コンポーネントのユーザー視点テスト |
| [@testing-library/jest-dom](https://testing-library.com/docs/ecosystem-jest-dom/) | DOM 向けカスタムマッチャー         |
| [@testing-library/user-event](https://testing-library.com/docs/user-event/intro)  | ユーザー操作のシミュレーション     |
| [Playwright](https://playwright.dev/)                                             | E2E                                |

> **Vitest を採用する理由:** ツールサービス WEB（フロントエンド）は React + Vite で構築しているため、Vite の設定・ES Modules・`import.meta.env` と相性がよい Vitest を標準のユニットテスト基盤とします。

---

## テストの実行方法

実装時に以下のコマンドを追加します。

| コマンド                     | 説明                                |
| ---------------------------- | ----------------------------------- |
| `pnpm --filter web test`     | ユニットテストを watch モードで実行 |
| `pnpm --filter web test:run` | ユニットテストを 1 回だけ実行       |
| `pnpm --filter web test:e2e` | E2E テストを実行                    |

> **補足**
>
> - **ユニットテスト**: `vitest.setup.ts` で `@testing-library/jest-dom` を読み込み、共通のテスト初期化を集約します。
> - **E2E テスト**: Playwright から開発サーバーを自動起動し、ツールサービス API（バックエンド）と認証 WEB（フロントエンド）の挙動はネットワークモックで制御します。

## ユニットテスト

### モック戦略

`fetch`・ルーティング・`AuthContext`・サイレントリフレッシュなどの外部依存は Vitest の `vi.fn()` / `vi.mock()` / `vi.stubGlobal()` で差し替え、コンポーネントやロジックを分離して検証します。

```ts
vi.stubGlobal('fetch', vi.fn());
```

`window.location`、`postMessage`、`message` イベント、iframe 作成などブラウザ API を扱う処理は、直接 DOM を操作するロジックを小さな関数に分離し、ユニットテストでは成功・失敗・タイムアウトを明示的に検証します。

### 優先度の目安

| 優先度 | 対象の例               | 検証観点                                                          |
| ------ | ---------------------- | ----------------------------------------------------------------- |
| 高     | `AuthContext`          | 認証状態の初期化、アクセストークン保存、ログアウト、再認証        |
| 高     | ルートガード           | `/` の保護、`/auth/callback` の中立ルート扱い、未定義ルートの扱い |
| 高     | SSO コールバック画面   | `code` 取得、`POST /auth/sso/verify` 呼び出し、成功・失敗時の遷移 |
| 中     | ダッシュボード画面     | ユーザー情報表示、ログアウトボタン、認証エラー表示                |
| 中     | API クライアント       | `Authorization` ヘッダー付与、401 時の再認証、1 回だけのリトライ  |
| 低     | 表示専用コンポーネント | 実質ロジックが薄い場合は省略してよい                              |

意図的に省略しがちな領域（スナップショットのみ、スタイルの細部など）は、変更コストと相談して範囲を決めます。

### 主要ケース

| 対象                 | ケース                                                                                                       |
| -------------------- | ------------------------------------------------------------------------------------------------------------ |
| `AuthContext`        | アクセストークンなしで起動した場合、未認証として扱う（起動時のサイレントリフレッシュは試行しない）           |
| `AuthContext`        | `refreshAccessToken()` 経由でサイレントリフレッシュ成功時、`POST /auth/sso/verify` の結果をメモリに保存する  |
| `AuthContext`        | サイレントリフレッシュ失敗時、未認証として扱い `/login` へ誘導する                                           |
| `AuthContext`        | ログアウト時、認証 WEB の `/logout?redirect=<ツール WEB の /login 完全 URL>` へ `window.location` で遷移する |
| SSO コールバック画面 | `code` がある場合、`POST /auth/sso/verify` を呼び出して `/` へ遷移する                                       |
| SSO コールバック画面 | `code` がない場合、[`/login`](./screens/login.md) へリダイレクトする                                         |
| API クライアント     | `401` 受信時にサイレントリフレッシュを試み、成功時は元リクエストを 1 回だけリトライする                      |

## E2E テスト

### 前提条件

Playwright は Chromium を主なテスト対象とし、ツールサービス WEB（フロントエンド）の開発サーバーを自動起動して検証します。ツールサービス API（バックエンド）と認証 WEB（フロントエンド）へのリクエストは `page.route()` でモックし、実 API への接続を回避します。

### 外部 API のモック

E2E テストでは、以下をテストごとに明示的にモックします。

| 対象                                           | モック内容                                                                                                |
| ---------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| `GET /auth/me`                                 | 認証済みユーザー情報、または `401 Unauthorized`                                                           |
| `POST /auth/sso/verify`                        | SSO 認可コード検証成功、または `401 Unauthorized`                                                         |
| 認証 WEB（フロントエンド）の `/login`          | 遷移先 URL の検証                                                                                         |
| 認証 WEB（フロントエンド）の `/silent-refresh` | `postMessage({ type: 'auth-code', code })` の送信、またはタイムアウト                                     |
| 認証 WEB（フロントエンド）の `/logout`         | `redirect` クエリ先へ `302` するなど、ツール WEB の [`/login`](./screens/login.md) へ接続できるようにする |

モック漏れがあると実 API へリクエストが飛ぶため、`setupAuthMocks()` のようなヘルパーへ集約して漏れを防ぎます。

### 主要ケース

| 対象             | ケース                                                                                                 |
| ---------------- | ------------------------------------------------------------------------------------------------------ |
| ダッシュボード   | 認証済みユーザーが `/` にアクセスするとユーザー情報が表示される                                        |
| ダッシュボード   | 未認証ユーザーが `/` にアクセスし、[`/login`](./screens/login.md) へ遷移する                           |
| ダッシュボード   | ログアウトボタン押下で認証 WEB の `/logout?redirect=…` へ遷移し、[`/login`](./screens/login.md) に至る |
| SSO コールバック | `/auth/callback?code=...` で検証成功時、`/` へ遷移する                                                 |
| SSO コールバック | `/auth/callback` に `code` がない場合、[`/login`](./screens/login.md) へ遷移する                       |
| SSO コールバック | `POST /auth/sso/verify` が `401` の場合、[`/login`](./screens/login.md) へ遷移する                     |
