## テストの種類

本プロジェクトでは **ユニットテスト** と **API 結合テスト** の 2 種類のテストを運用します。

| 種類           | 概要                                                                                    |
| -------------- | --------------------------------------------------------------------------------------- |
| ユニットテスト | サービス関数・認証ガード・認証 API クライアントを依存関係からモック分離して単体検証する |
| API 結合テスト | FastAPI アプリに対して HTTP リクエストを送り、エンドポイント全体を結合検証する          |

---

## 使用フレームワーク

| ツール                                                               | 用途                                                               |
| -------------------------------------------------------------------- | ------------------------------------------------------------------ |
| [pytest](https://docs.pytest.org/)                                   | テストランナー・アサーション                                       |
| [pytest-mock](https://pytest-mock.readthedocs.io/)                   | モック                                                             |
| [FastAPI TestClient](https://fastapi.tiangolo.com/tutorial/testing/) | API 結合テスト                                                     |
| [httpx](https://www.python-httpx.org/)                               | 認証 API（バックエンド）への HTTP クライアント、および非同期テスト |

> **補足:** Phase 1-b では DB を使わない認証連携 API が中心のため、実 DB を使う E2E テストは導入しません。ツールデータを永続化する Phase 2 以降で DB が入った段階で、DB を含む E2E テスト方針を追加します。

---

## テストの実行方法

実装時に以下のコマンドを追加します。

| コマンド                   | 説明                                             |
| -------------------------- | ------------------------------------------------ |
| `uv run pytest`            | ツールサービス API（バックエンド）のテストを実行 |
| `uv run pytest tests/unit` | ユニットテストのみ実行                           |
| `uv run pytest tests/api`  | API 結合テストのみ実行                           |

> **補足**
>
> - テスト依存は `apps/api/pyproject.toml` の開発依存に追加します。
> - 認証 API（バックエンド）への通信はテスト内でモックし、実 API への接続を回避します。

## ユニットテスト

### モック戦略

認証 API（バックエンド）への HTTP 通信、JWT 検証、環境変数などの外部依存はモックに差し替えます。サービス層では入力に対して正しい外部 API 呼び出しが行われること、外部 API の結果を正しく変換すること、エラーを適切な HTTP 例外へ変換することを検証します。

| 依存                                             | モック方針                                                   |
| ------------------------------------------------ | ------------------------------------------------------------ |
| 認証 API（バックエンド） `/auth/exchange/verify` | `httpx` クライアントまたはラッパー関数をモック               |
| JWT 検証                                         | 正常なクレーム、不正署名、期限切れ、`aud` 不一致を返すモック |
| 環境変数                                         | `monkeypatch` でテストごとに設定                             |

### レイヤー別の検証観点

| レイヤー   | 検証観点                                                                                                          |
| ---------- | ----------------------------------------------------------------------------------------------------------------- |
| Router     | リクエストボディ・ヘッダーを受け取り、サービスに正しい引数を渡すこと / サービスの戻り値をレスポンスとして返すこと |
| Service    | 認証 API（バックエンド）を正しい payload で呼び出すこと / 失敗時に適切な例外を返すこと                            |
| Auth Guard | `Authorization` ヘッダーを検証すること / 署名・有効期限・`aud` を確認すること / ユーザー情報を取り出すこと        |
| Settings   | 必須設定が揃っている場合は正常に起動すること / 必須設定が欠落している場合にエラーになること                       |

### 主要ケース

| 対象             | ケース                                                                           |
| ---------------- | -------------------------------------------------------------------------------- |
| SSO 検証サービス | SSO 認可コードを認証 API（バックエンド）の `/auth/exchange/verify` へ送信する    |
| SSO 検証サービス | 認証 API（バックエンド）が `200 OK` を返した場合、`accessToken` と `user` を返す |
| SSO 検証サービス | 認証 API（バックエンド）が `401 Unauthorized` を返した場合、`401` を返す         |
| SSO 検証サービス | 認証 API（バックエンド）との通信に失敗した場合、`500` を返す                     |
| Auth Guard       | 有効なツールサービス向けアクセストークンからユーザー情報を取り出す               |
| Auth Guard       | Authorization ヘッダーがない場合、`401` を返す                                   |
| Auth Guard       | アクセストークンが期限切れの場合、`401` を返す                                   |
| Auth Guard       | `aud` がツールサービス向けでない場合、`403` を返す                               |

## API 結合テスト

### 前提条件

FastAPI の `TestClient` を使い、アプリケーションに対して実際の HTTP リクエストを送信します。認証 API（バックエンド）への外部通信はモックし、テストが外部サービスの状態に依存しないようにします。

> **補足:** テストでは本番同様のバリデーションを通すため、FastAPI アプリ本体を import して `TestClient(app)` を作成します。

### 外部 API のモック

API 結合テストでは、以下をテストごとに明示的にモックします。

| 対象                                                  | モック内容                                             |
| ----------------------------------------------------- | ------------------------------------------------------ |
| 認証 API（バックエンド） `POST /auth/exchange/verify` | SSO 認可コード検証成功、`401 Unauthorized`、通信エラー |
| JWT 検証                                              | 有効なクレーム、期限切れ、`aud` 不一致                 |

モック漏れがあると実 API へリクエストが飛ぶため、`mock_auth_exchange_verify()` や `make_access_token_claims()` のようなヘルパーへ集約して漏れを防ぎます。

### エンドポイント別の主要ケース

#### POST /auth/sso/verify

| ケース                                  | 期待結果                                 |
| --------------------------------------- | ---------------------------------------- |
| 有効な `code` を送信する                | `200 OK` と `accessToken`, `user` を返す |
| `code` が空文字                         | `400 Bad Request`                        |
| `code` が欠落                           | `400 Bad Request`                        |
| 認証 API（バックエンド）が `401` を返す | `401 Unauthorized`                       |
| 認証 API（バックエンド）との通信に失敗  | `500 Internal Server Error`              |

#### GET /auth/me

| ケース                                             | 期待結果                  |
| -------------------------------------------------- | ------------------------- |
| 有効なツールサービス向けアクセストークンを送信する | `200 OK` と `user` を返す |
| Authorization ヘッダーがない                       | `401 Unauthorized`        |
| Bearer 形式でない                                  | `401 Unauthorized`        |
| アクセストークンが期限切れ                         | `401 Unauthorized`        |
| `aud` がツールサービス向けでない                   | `403 Forbidden`           |
