## 基本方針

- Python 3.11 以降を前提に、型ヒントを明示します。
- コメント・docstring・エラーメッセージ・テスト名は日本語を基本にします。
- 実装は既存のレイヤー構成（Router / Service / Client / Dependency / Auth / Schema / Core）に沿って配置します。
- 新しい抽象化は、責務分離や重複削減に明確な効果がある場合だけ追加します。

## レイヤー責務

- `app/main.py` は FastAPI アプリの生成、共通設定、ルーター登録、ヘルスチェックに限定します。
- `app/routers/` は HTTP パス、メソッド、依存注入、レスポンスモデルの定義を担当します。
- `app/services/` はユースケースを組み立て、外部クライアントの結果を API の応答モデルへ整えます。
- `app/clients/` は外部サービスとの通信と、その通信失敗の変換を担当します。
- `app/dependencies/` は Router に差し込む共通処理を置きます。
- `app/auth/` は JWT など認証の下位処理を置きます。
- `app/schemas/` は Pydantic モデルを置き、リクエスト・レスポンスの形を明示します。
- `app/core/` は設定やアプリ全体の基盤処理を置きます。

## docstring

- モジュール、公開関数、公開クラスには短い docstring を付けます。
- 引数と戻り値の説明は、FastAPI や Pydantic のお作法で自明なものを除き、認証、外部通信、リトライなど読み手の理解に影響する場合に書きます。
- docstring は「何をするか」だけでなく、責務・意図・境界が伝わる表現を優先します。
- 処理コメントは、認証状態の分岐、トークン再取得、タイムアウト、リトライなど意図が重要な箇所に記載します。
- テストにも、モジュール、fixture、ヘルパー、テストクラス、テスト関数の意図を短く書きます。

## エラー処理

- API に返す失敗は `fastapi.HTTPException` で明示します。
- 認証失敗は `401 Unauthorized`、権限・audience 不一致は `403 Forbidden` を使い分けます。
- 外部通信失敗や予期しない上流失敗は `500 Internal Server Error` に変換します。
- 外部通信例外は `raise ... from exc` で原因を保持します。

## テスト

- ユニットテストは `apps/api/tests/unit/`、API 結合テストは `apps/api/tests/api/` に置きます。
- テストファイルは `test_*.py`、テスト関数は `test_*` とします。
- 環境変数は `monkeypatch` でテスト用に固定します。
- 外部 API 通信は `pytest-mock` でモックし、実サービスに接続しません。
- API 結合テストは `TestClient(app)` でルーティング、依存注入、レスポンス生成まで確認します。
